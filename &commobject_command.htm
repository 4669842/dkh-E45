&CommObject

Syntax    

[&CommObject type = "<InterfaceID>",action = "<FunctionOfCommand>",names = "<1st name|2nd name|3rd name...>",values = "<1st value|2nd value|3rd value ...>"]

Example Scripts

Description

Sends commands to iManage Work and can also be used within the Document Bundling integration.

Integrated documents can be accessed and opened via Visualfiles scripting using the &CommObject command. See example 8 for more information.

Prerequisites

The iManage Work integration must be fully defined in Visualfiles. For more information go to Set up the Link Between iManage Work and Visualfiles.

Parameters

action = "<FunctionOfCommand>"

The action parameter defines the function of the &CommObject command. The available actions are:

AddProtectiveMarkings

Adds markings to a Class, Folder or Document in xml

DeclareRecord

Declare item as a record

CreateFolder

Create a folder

CancelCheckOut

Close the document without saving changes

CheckInDoc

Close the document and save changes

CheckOutDoc

Open the document for editing

DeleteDoc

Delete the document

GetAccess

Update access control lists in xml. See example 8.

GetDoc

Open the document read-only

GetLookup

Constant value

Note: You can specify this parameter to display a iManage Work lookup table and retrieve a value from it.

GetMetaData

View Metadata associated with the item

EditMetaData

Edit Metadata associated with the item

OpenDoc

Access and open integrated documents

PlaceDocument

Adds a document to a particular file plan

StoreDoc

To save the document

SearchDoc

Open search dialog

Logout

Kill a session

names = "<1st name|2nd name|3rd name...>"

The name of the parameter passed to the action. Names must be delimited by a pipe symbol '|'. The names parameter and number of names required, is dependant on the action being called.  Some actions, such as Deletedoc, require no names or values.

In the case of iManage, the names parameter is used to list the profile details being passed: Author, Operator, Class, Description, etc.

type = "<InterfaceID>"

A three-digit code identifying the interface used.

values = "<1st value|2nd value|3rd value...>"

A list of values associated with the names delimited by a pipe symbol (|).  The number and format of the values required is dependant on the action and name supplied.  For each name specified, a corresponding value must be supplied - even if that value is blank (where the default value will be used, if allowed).  Some actions and names will require valid, specific values.

The values can be separate: as text enclosed in quotes; contained in a temporary field; or a concatenation of temporary fields.

Example Scripts

1 Using the deletedoc action with iManage Work

[&CommObject type = "IMA",action = "deletedoc"]

2 Using the storedoc action with iManage Work

This script will save a document and set the class, author and operator profile details as doc, a.lewis and c.white respectively. As no dbname is specified, the default value will be used.

[&CommObject type = "IMA", action = "storedoc",names = "class|author|operator|dbname", values = "doc|a.lewis|c.white|"]

3 Using the storedoc action with iManage Work and a temporary field containing the values

This script will save a document and the class, author, operator and dbname profile settings are read in from a temporary field.

[&CommObject type = "IMA",action = "storedoc",names = "class|author|operator|dbname", values = TF01]

4 Using the checkoutdoc action with iManage Work

This script collects the username and password required by the integration for performing actions such as checking out documents.

[&Assign DM06 = ""]
[&CommObject type = "IMA",action = "checkoutdoc",names = "user|password", values = "n.person|Abc123"]

5 Using the createfolder action with Meridio

This script creates a folder on the Document Management system to store the documents.  

[&CommObject Type="MER" , Action="CREATEFOLDER" , Names="FOLDER|PARENT|COMMENT" , Values = TF300 + "|" + TF400 + "|FOLDER CREATED BY " + US01]

6 Use the destination parameter to specify two folders to which emails will be saved in iManage Work

A semicolon is used to separate the names of two folders to which email will be saved.  Sub-categories can be specified using a "/", for example, "Emails;Letters/Sub Category"

[&ASSIGN DM06 = ""]

[&IF US01="SARAHBELL"]

   [&Assign TF03="S.BELL"]

[&Else]

   [&Assign TF03=""]

[&EndIf]

[&assign tf01 =  "Emails;Letters|DOC|"+TF03+"|"+TF03+"|no|ansi|another test desc"]

[&Assign TF02=""]

[&COMMOBJECT TYPE="IMA" , ACTION="STOREDOC" ,NAMES="destination|CLASS|AUTHOR|OPERATOR|Profile|type|description" , VALUES=tf01]

7 iManage custom2 table parentalias

The following command filters the custom2 table where the parentalias field = test:

[&CommObject type = "ima",action = "getlookup",names = "parentalias|table|destination|profile",values = "test|custom2|Roo02|no"]

Within iManage, the tables custom2, custom30 and subclass require that a parentalias is specified.

The element parentalias can be specified. It is a parent value from which the table results are built (subclassing).

The element table is the name of the iManage table.

The element destination is the Document Category in which the document/email is to be stored.

8 Using the OpenDoc action

Documents in History can be viewed from running an Agenda item but not edited.

Documents in History can be checked out and edited from running an Agenda item. They are marked as Checked Out within iManage Work and within the case History.

[&Assign LF03="1139"] **history number (HS26)

[&Assign LF01="history-ID|Mode"] **names

[&Assign LF02=LF03 + "|edit"] **values (Mode can be EDIT or VIEW)

[&COMMObject Type="ima", Action="OpenDoc", Names=LF01, Values=LF02]

[&If DM06<>""] **error

[&Message DM06]

[&EndIf]

9 Update Access Control Lists in Meridio

To update Access Control Lists in Meridio you can run the following function:

[&CommObject type="MER" , action="GETACCESS" ,names="PARENTID|OBJECTID|PARENTTYPE|OBJECTTYPE",values= TF05 ]

This has been wrapped up in the attached Script Mer-Access. This functionality has been in the product for some time.

The purpose of the function is to copy the ACL from one object to another.

Add  the following code to the end of the ADD document script:

[&Call SYSTEM.MER-ACCESS, input = MER02 , input = DM09 , input = "Folder" , input = "Document"]

This copies the ACL from the parent folder onto the Document (only required if inheritance does not occur between the folder and document)

At the start of the Declare Record script you can add:

[&Assign LF09 = DM09] /* Store the original document ID */

At the end of the Declare Record script you can add:

 [&iF LF09 <>  DM09] /* Only updates if Declare and Delete used as IDs will be different */

[&Call SYSTEM.MER-ACCESS, input = LF09 , input = DM09 , input = "Document" , input = "Document"]

[&Endif]

This copies the ACL from the deleted document to the new record.

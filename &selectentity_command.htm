&SelectEntity

Syntax    

[&SelectEntity {type = <EntityTypeCode>|role = <RoleCode>|int-code = <EntityInternalCode>}]

Example Scripts

Description

Selects a specific entity from the database.

Selecting an Entity from a Relation Field

It is possible to build a dynamic process that runs multiple workflows with a small amount of generic scripting with the workflow decisions and data held at entity level. You can select an entity using the data held in a relation field on a UD screen. See example 6.

Parameters

button1 = "<ButtonXText>"

Assign text to buttons displayed in the Finder. The &SelectEntity command can display a dialog with up to five buttons: the buttons are named button1, button2, button3, button4 and button5. To have one of these buttons show on a screen, say button3, add button3 = "button 3 text", to your command.

code = <ContentOfXXX>,field = "<FieldName>"

The field and code parameters are used together to identify an entity by the contents of a specific field. The referenced field must be an indexed field.

default = "<ButtonNumber>"

Assign a button as the default.

The default button has the focus applied and its value will be assigned if a user simply presses Return on a screen. If the default parameter is not used, focus will apply to button1.

fetch = "<{Numeral|All}>"

Enables a developer to control how many records will be returned to the Finder.

field = "<FieldName>"

The field and code parameters are used together to identify an entity by the contents of a specific field. The referenced field must be an indexed field.

code = <ContentOfXXX>,field = "<FieldName>"

int-code = "<EntityInternalCode>"

Select an entity by specifying the entity's internal code (EN56).

Note: The int-code parameter must be included in the &SelectEntity command when scripting for Visual Exchange or the following error will be returned:
'User Entity selection not allowed when running script in batch mode'.

name = "<ContentEN02>"

When used with the type parameter, allows the selection of an entity by specifying the entity's name (EN02).

noadd

Remove the Create button from the Finder, preventing users from creating new items when selecting an entity.

ref = "<TextString>"

Gives a reference to the selected entity.

The ref parameter provides a quick reference to the selected entity that can be used throughout the remainder of the script.

response = <Field>

Specifies the location to which the user response should be stored.  If the user double clicks on their selection or highlights a finder entry and presses return, response will equal "GO".  If the user clicks on the close window button in the top right hand corner, or presses escape, response will equal "CANCEL".  If any of the user-defined buttons are clicked, response will equal "BUTTONX".

role = "<RoleCode>"

Restrict the selection of an entity to only those with a specific role.

title = "<Title>"

Assign a title to the dialog box.

type = <EntityTypeCode>

Specify the entity type code.

If this parameter is not included, the command will return the error Entity type not specified.

If the default owner is a role, then its entity type may be selected. This entity then becomes the default owner (only enabled in system scripts).

The concept of 'default owner' is fundamental to scripting in Visualfiles. The default owner is the file, entity or role in which a script is located at a given point in time during the processing of the script. The default owner changes as the script moves through the elements in the system.

System scripts, by nature, have no default owner and will inherit ownership of the last accessed component as the script progresses.

For example, in very simple terms, say you have a  system script that starts at an entity then references a role and, finally, selects a file.

As the script is processed, and the role is referenced, ownership is transferred to that role and it becomes the default owner. Similarly, as the script is processed and the file is selected, the file will be the default owner.  

The default owner  for all other scripts will be the  file,  entity or role   where the processing starts, but as with system scripts this can be changed using the &With command.

type = "user",user = "<UserID>"

Select an entity related to the specified user.

The user parameter can only be used in conjunction with a type = "user" parameter/value combination that references the internal code (US13) to identify the entity associated with the logged-in user. When used in this combination, this parameter can be used to select a user other than the user currently logged in.

user = "<UserID>"

The user parameter can only be used in conjunction with a type = "user" parameter/value combination that references the internal code (US13) to identify the entity associated with the logged-in user. When used in this combination, this parameter can be used to select a user other than the user currently logged in.

The user parameter can be used in conjunction with user-defined lookup fields, internal codes or the user ID to search for the associated entity for any file holder.

Example Scripts

1 Entity type

This code launches a Finder window from which the user can select a Commercial entity.

[&SelectEntity type = "COMM"]

2 Using Temporary fields

Having assigned the required entity type to a temporary field, the example below selects an entity of that type and assigns a reference to it.

[&Assign TF01 = "INDI"]

[&SelectEntity type = TF01,ref = "selectedEntity"]

3 name parameter

Selecting an entity called Mr Bob Smith with the role "agent".

[&SelectEntity role = "Agent",name = "Smith, Bob Mr",ref = "MrSmith"]

4 int-code parameter

Where TF01 has earlier been assigned the internal code of the relative entity, the example below will re-select that entity.

[&SelectEntity int-code = TF01]

5 button parameter

This example specifies the text for three buttons and stores the user response to a temporary field for reference.

[&SelectEntity type = "COMM",button1 = "Cancel",button2 = "New Commercial",button3 = "Select",button4 = "Cancel selection",response = TF03,ref = "chosenComm"]

6 Build a dynamic process running multiple workflows

You can build a dynamic process that runs multiple workflows with a small amount of generic scripting with the workflow decisions and data held at entity level. You can select an entity using the data held in a relation field on a UD screen.

In this example OPY16 is a relation field.

Assign LF01=”OPY16(EN56)”

Assign LF01=fieldvalue(LF01)

Selectentity int-code=LF01

&Include

Syntax    

[{&Include}<DocumentName>|[Source=<"SourceName">]...]

Example Scripts

Description

Merges one Visualfiles document or script into another.

&Include is commonly used to include small header or footer documents into larger generated documents to minimise coding. When information in one of these documents changes, it is updated in only one document.

The prescript of any included document will not execute: this scripting must be included in the generated document.

Generated documents, containing included documents, will not reflect changes to those included documents until the agenda containing the documents has been recompiled.

&Include can also be used to chain a number of documents together into a single agenda item. Chains can be made more flexible by including conditional statements, so that individual documents in the chain can be made optional.

Add-ins

In an add-in document it is possible to include a document that is located on the file that the add-in is linked to, or another add-in linked to that file, using the source parameter. However, in order to ensure that the database fields in the source document work when it is run, it is necessary to include some additional scripting to the source document.

If the document being included is a file level document, all database fields must be prefixed with the owner parameter:

owner.ABC01

Where ABC01 is the database code.

Note: Adding the owner parameter to the source document on the file does not affect how the source document compiles or runs elsewhere in the file.

If the document being included is from another add-in, all database fields must be prefixed with the add-in code:

addincode.ABC01

Where addincode is the unique code of the source add-in and ABC01 is the database code.

Note: Adding the owner parameter to the source document on the file does not affect how the source document compiles or runs elsewhere in the file.

Parameters

Source

Specify the location of the called document.

<DocumentName>

Specify the document to be included within another document.

Example Scripts

1 Including a header document in a letter

This script inserts the document L- GEN-HD into a letter to a client in a conveyancing case: the document is included at the point where the &Include command is located in the letter.

[&Include L-GEN-HD]

Note: The document L- GEN-HD is a header document that contains the following code.

[CLIENT_CLIENT.EN05]
[CLIENT_CLIENT.EN34]
[CLIENT_CLIENT.EN35]
[CLIENT_CLIENT.EN36]
[CLIENT_CLIENT.EN37]
[CLIENT_CLIENT.EN38]

Date: [DATE:ds("dw mw yc ")]

Our Ref: [FE02]/[US03]/[MT05]

Dear [CLIENT_CLIENT.EN05(3)]

re: Your Purchase of Property [PRO01][PRO02][PRO03]

2 Creating a chain of documents for an agenda

In a conveyancing file, the first letter sent to the client, S-EN-LET1, may be accompanied by a fixtures and fittings form, S-FFC, a form of authority, S-FOA, and a client information form, CCIF. Rather than selecting four different agenda items, a single document is set up that includes all four documents. A hard page break has been inserted between each document included in the chain.

[&Include S-EN-LET1]

--------------------------page------------------------------

[&Include S-FFC]

--------------------------page------------------------------

[&Include S-FOA]

--------------------------page------------------------------

[&Include CCIF]

3 Include the document Header from the parent directory

[&Include ..\Header]

4 Source parameter

When the source parameter is used the called script is retrieved from the specified location. But when it is run, the script behaves as if it exists in the location it is called from. This example explains this using two add-ins:

Administration ADMINISTRATIONAA

Sale Instructions SALEINSTAA

The following document exists on the ADMINISTRATIONAA add-in:

Add-in Details L-ADMINISTRATIONAA-DETAILS

The document contains the following code:

Add-in Name: [AI03]

If this document is run from the ADMINISTRATIONAA add-in the result would be:

Add-in Name: Administration

If it is run for the SALEINSTAA add-in using the include command with the new source parameter:

[&Include S-ADMINISTRATIONAA-DETAILS, source="ADMINISTRATIONAA"]

Then the document is retrieved from the ADMINISTRATIONAA add-in but when it is compiled and run it behaves as if it exists on the SALEINSTAA add-in so the result is:

Add-in Name: Sale Instructions

5 Call an agenda level document from an add-in using the source parameter

[&Include D-GEN-BLANK, source=”MATA”]

Includes a file-level document header inside an add-in document.

This script includes the D-GEN-BLANK from the MATA application agenda when it is run from the add-in agenda.

Note: Ensure that you have compiled the add-in and that the document is coming from and the add-in or application it is being included from.

6 Call Add-in documents from Agenda level documents on the file using the source parameter

The source= parameter has to be used to identify the add-in the document is on.

[&Include DOC1,source=”Prospectaa”]

&Request

Syntax    

[&Request URL = URLaddress]

Example Scripts

Description

The &Request command is used to call a web service. The command takes the contents of an XML file, sends this information to the specified URL and receives the response. This functionality is achieved by packaging the XML document as a SOAP message and invoking the specified SOAP action. The format of the XML that is sent and received and the SOAP actions supported will be dependent on the web services being used and will be documented by the service provider.

The &Request command can be used in conjunction with database fields, including temporary fields.

Parameters relating to the use of QAS (Quick Address Search) software's web services are given towards the end of the list of parameters below.

Note: HTTPS in not supported by &Request but you can make use of the &Script command to achieve this; &Script enables developers to integrate with third party products by calling and running external scripts. HTTPS can also be supported indirectly through a third party server.

Closing connections

Progress will stop waiting for a response from a &Request command when one of the following occurs:

  • The server closes the connection.

  • The size of the response matches the Content-Size passed as part of the response header

  • The End-Element is reached.

Whichever condition is satisfied first will close the connection. Where none of these conditions are met, Visualfiles may hang and crash the user out of the system.

Note: Some parameters relate to using QAS (Quick Address Search) software's web services.

Parameters

attach-element = <Element>

Element of the XML data with which to store the document.

attach-file = <FileName

The file to attach as part of the request (file is A group of similar encoding schemes that represent binary data in an ASCII string format by translating it into a radix-64 representation. The Base64 term originates from a specific MIME content transfer encoding.B64 encoded automatically).

Binding = <binding name>

Alternative to the parameter "service", depending on the web service - see example 4.

Note: Relates to QAS.

BindingNamespace = binding

Alternative to "ServiceNamespace when the parameter binding is used, depending on the web service - seeexample 4.

Note: Relates to QAS.

content-type = "<ContentType

Used to explicitly define a content type such as “application/x-www-form-urlencoded rather than using the default text/xml.

end-element = "<LastNode

Used to define the last expected node of an XML response so that Progress can detect if the whole of the response has been received. This is only required if the server sending the response does not send Close Connection. It is recommended that developers use the End-Element parameter to force a connection to close rather than relying on a response from the remote server to close the connection. Where the connection is left open, Progress does not know when to stop waiting for a response: adding an End-Element parameter enables Progress to parse the response and quit as soon as it hits the end tag.

full-response

Return all of a Web Service response to the XL field rather than just the Body-Text.

hdr-xml =

Specifies a block of XML to place in the SOAP header.

no-element

Don't include the attach element tags, just the document.

operation = <operation>

Determines the operation in the Web Service to call.

output =

The XL field code for the web service response.

output-element = <LocationOfDocumentInResponse

The location of the encoded document in the XML response.

output-file = <LocationForResponse

The file and location in which to store any document passed back in the response (file is B64 unencoded automatically).

port = <port name>

The port being used to connect to the web service.

Note: Relates to QAS.

post-process

Used to ensure that where the response being received from a web service command is incomplete, the response will be stored and processed once the socket connection is closed.

request = <XMLDataOrFieldThatHoldsIt

Holds the XML data or else specifies the field that holds the XML data. This parameter may be set equal to the XML data itself, in which case the data should be enclosed in speech marks, otherwise, it should be set to the name of the field in which the XML data is held, for example, TF01.

response = <response>

Used to specify the XL field to hold any response from the call.

service = <service name>

The name of the service being called.

Note: Relates to QAS.

ServiceNamespace =

Namespace of the service being called.

Note: Relates to QAS.

soap

This parameter specifies that the web services request is a SOAP package.

SoapAction = "<ActionOrFieldThatHoldsIt

Specifies the action to be performed: this may be the action itself, in which case the action should be enclosed in quotes, or, it may be a temporary field that currently holds the action, for example, TF02.

SoapEndpoint = URL Endpoint

Specifies the end point of the web service.

Note: Relates to QAS.

SoapEndpointPassword = Password

Password for the web service end point.

Note: Relates to QAS.

SoapEndpointUserid = User id

User ID for the web service end point.

Note: Relates to QAS.

soap-version = <SoapVersion

Soap version to use.

TargetNamespace = target namespace

Note: Relates to QAS.

URL = "<URL>"

Specifies the URL to which the web services request should be directed. This address should be written in full and be enclosed in double quotation marks. The URL parameter may be set equal to a database or temporary field that holds the URL.

WSDL = <WSDL-Document>

The web service being called.  At the very least only the WSDL parameter maybe specified with all the other parameters optional.

Note: Relates to QAS.

WSDL-Password = <password>

Password for the web service being called.

Note: Relates to QAS.

WSDL-Userid = <user-id>

User-id for the web service being called.

Note: Relates to QAS.

4 &Request “Connection" or “binding" parameters

This method is considered the recommended way of calling web services from Progress. The 'connection' parameters for the &Request command are wrappers around the actual Progress parameters to provide a generic solution.

This example shows the different options for binding to the WSDL(Web Services Description Language). At the very least only the WSDL parameter may be specified with all the other parameters optional. The example shows two methods of binding. The method of connection is purely dependent on the Web Service.  

All other 'locator' parameters to do with &Request are not used when this method is chosen (such as URL= &SOAPACTION=).

Example 1

WSDL=wsdl-document [WSDL-Userid=user-id [WSDL-Password=password]]

  [Service=service-name [ServiceNamespace=service-namespace]][Port=port-name]

  [SOAPEndpointUserid=user-id [SOAPEndpointPassword=password]]

[TargetNamespace=targetNamespace]

Example 2

WSDL=wsdl-document [WSDL-Userid=user-id [WSDL-Password=password]]

  [[Binding=binding-name [BindingNamespace=binding-namespace]] SOAPEndpoint=URL-endpoint]

  [SOAPEndpointUserid=user-id [SOAPEndpointPassword=password]]

[TargetNamespace=targetNamespace]

5 Call a web service

[&Request WSDL="http://awebservice.com/somewhere.wsdl", SERVICE="ProWeb", PORT="QAPortType", OPERATION="DoSearch", REQUEST=TF01, RESPONSE=XL01]

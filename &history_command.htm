&History

Syntax    

[&History "<MessageText>"]

Example Scripts

Description

Creates a new history item and saves it to the history of a file.

Parameters used in conjunction with this command include functionality such as saving documents, charging for time, setting the date, setting the category, activating document versions, setting access level, recording timer information, etc.

An &History command includes message text (contained in double quotes " ")  that is posted to the History screen of the selected file along with the date of the action. This message can contain up to 259 characters but only 32 are displayed on the File History screen. If the required history message is longer than 50 characters, then this extra information may be added to the history item as additional text.

When saving an email into file history you can edit the Import Document Script in the application editor to override the standard behaviour; for example, to prompt users for additional information or to create a reminder.

Document Bundling

During the save to file history process when using Document Bundling the Import Document Script of the file is run which incorporates the &History command and relevant parameters, which are used to create the required document bundle versions. These parameters are auto, auto-save, docid, keydate, master and versions. See examples 15-20.

The parameters list below is organised into General Parameters and Email Parameters.

General Parameters

access = <AccessLevelCode>

Attaches a condition to the history item: where a user has an access level lower than the one required, they cannot view the history item.

activity

Enables an activity type to be associated with the history item and allows time to be recorded against that item.

Note: The Activity type must already be setup in Administration using Activity Codes.

The charge is applied against the Active File Holder specified by the fee-earner parameter, or subsequently selected in the History Item dialog box.

amend

Displays the History Item dialog so that users can add or amend some of the history item details including Time units.

Once the agenda item containing the &History command has been actioned, the History Time Item window is displayed.

This dialog box is not displayed if the File Holder has Time Sheets enabled; the item is saved directly to their time sheet.

Note: When the [&History ,amend] command is used the history item added is date for the history item is checked. If the date for the history item is greater than one year old the user is asked to confirm this is correct, 'Date entered is greater than 1 year old. Is this correct?'

auto

Automatically creates a new version of a document on closing when used in conjunction with versions.

This parameter is used in Document Bundling.

auto-save

Automatically saves the document when a DocId is also supplied. Prevents the version dialog or save prompt from appearing. The history item is saved silently.

This parameter is used in Document Bundling.

billable = "{Y,Yes,True,X}"

Allows a user to specify that the history item should be excluded from bills when they are compiled.

call = <scriptname>

This parameter enables you to  specify a script to run when a document has been closed by a user. It can only be used to capture information that is to be stored as metadata after the document has been edited by the end user. Although a postscript is provided with each document, they are run prior to the document being displayed. See example 23.

Note: This parameter can only be used in conjunction with documents run from the agenda.

category = "<CategoryCode>"

Sets the history item as being of a particular category when used in conjunction with the save, scan or read parameters. The category can then be used as a means of filtering history items. A maximum of 15 categories can be assigned to a history item.

The category does not need to be setup before it is specified in the &History command.

Use this parameter to store the generated document in a specified existing category and create the category entry if required. Multiple categories and multi-layered categories can be created; see also &Category. See examples 4 - 6.

CSB-Stage="<csb stage>"

Used to specify the case stage budgeting stage.

date ="<dd/mm/yyyy>"

By default, the history item is created with today's date. This parameter is used within an &History command to set a specific date (by supplying the date in dd/mm/yyyy format). Also used to prompt the user for a date by not specifying a date after the date parameter.

docdate = "<dd/mm/yyyy>"

Allows a user to specify a date which is recorded as the amended date for the history item.

docid =

The ID of the history document to save. Indicates that a previous version of a history item is being saved.

This parameter is used in Document Bundling.

This parameter enables you to use the &History command to save a version of a document rather than using it to create a new item; the DocID parameter lets you specify which Item to update.

fee-earner = "<FileHolder/FeeEarner>"

Specifies the fee-earner for a history item. Use this parameter to override the allocated fee earner for the history item; by default the allocated the fee earner for the matter is recorded.

Important

Used to auto-flag history items as 'Important' when they are created, and displayed on the History screen where you can filter items on this status. It is used to record significant stages in the life cycle of a file. For more information go to HS46. See example 7.

index

This parameter is used in a Profile Script to force Visualfiles to profile documents. When it is included, the custom properties FileCode and FileHolder are automatically added to the document. See Microsoft Index Server overview for more information.

Note: This parameter is only relevant if the Microsoft Index Server integration is activated.

KeyDate = "<date>"

Add a document date to the history item.

This parameter is used in Document Bundling.

level = UserAccessLevelCode

Where a user has a hierarchy level lower than the one specified, they cannot view the history item.

location = "<FilePath>"

Allows a user to specify an alternative FILEPATH that overrides the default file location for history documents. Used where distributed document locations are set up. Enables users to override a central location and save to a local drive.

[&History 'Description for History', save, location = ""]

This script prompts the user to enter text into the history description. The user is then prompted to select a file location from the tree view which is initialised to the appropriate file location for the user. This is the best method for users who normally save documents to their local file-server, but on occasion would want to save the document elsewhere.

[&History 'Description for History', save, location = "HAR"]

This script also prompts the user to enter a history description, then saves the document into file location HAR. If HAR is a valid location, the user is not prompted to select a location.

master

Indicates that the item is a master version. Used in conjunction with versioning.

This parameter is used in Document Bundling.

metadata

Used within a document/form to attach metadata to the history item. Used in combination with the save parameter.

If this parameter is used when saving documents to history, the metadata screen is displayed after the document has been edited to enable the user to enter details in the configured metadata fields. If the document is being created from a robot or in batch mode then the metadata screen is not displayed. See example 21.

This parameter is useful because the &Metadata command does not work in this situation, because the document would not have been created when the &Metadata command was processed.

noconfirm

The user is not prompted for confirmation before the item is saved to file history.

This parameter is used in conjunction with Time Recording - Real Time Capture against a history item. If Time Recording is not used, the user is prompted to choose to post the time recorded using the History Item Confirmation dialog.

noedit

Ensures that users viewing the agenda item from the Document Review cannot edit the document or form.

When this parameter is selected, the warning message 'This document is read only. Any changes will not be saved' is displayed before the item is opened. The item is set as read-only in the history item properties.

no-warn

Suppresses either of the two error messages that can be displayed when using this command. The error messages are a warning regarding the date, or a warning that the grade being used has not been flagged as Selected.

preview

This parameter enables documents to be run and previewed but not stored, when a Document Management SystemDMS such as iManage Work is switched on.

Note: This parameter is only relevant when the iManage Work integration is activated.

read = "<Filename>"

This parameter is used to automatically open a web browser to select a file. Alternatively, if the file name is supplied in the script by assigning the file name, and the read parameter references this field, no file selection browser is displayed, see example 25. When the &History command is run, the original document is copied to the Visualfiles directory structure.

When you drag and drop documents onto a file and the Import Document Script is used, ensure that the read parameter is specified in order to attach the imported documents to the file, see example 24.

Note: Read-only documents can be imported into file history.

,rtc

This parameter improves accuracy when recording timer information on files and delivers more streamlined document production process to the user. The ,rtc parameter enables any timer information on a file to be passed from the Real Time Capture module to the History screen where document and time information are combined in a single entry. This prompt can be used in documents and standard agenda scripts. See example 12.

Note: This parameter is used to control the Real Time Capture timer.

Ensure that the amend parameter is added whenever the rtc parameter is included in an &History command. When the document is closed the History Time Item dialog opens and the time for the document is posted, if a timer was running when the document was opened. A new timer then starts automatically.

Note: The time posted is rounded up to the next standard unit as defined for the Activity code in Administration.

save

Used within a document/form to save the document/form to the history item.

Note: If documents are produced in a Chain, only one save is performed. This saves the full sequence of documents as one item.

scan

Prompts the scanning of a document or form into a file history.

storename = "<DocID>"

Allows a user to specify a document ID to replace the system-generated document ID.

Note: This parameter is only used when iManage Work integration is switched on.

time="<hh:mm:ss>"

Specify posting time. Use this parameter to override the posting time.

Specifying the time parameter alone prompts the user to enter the time. 24-hour format and seconds cannot be specified (the value is 0).

Where a history item already has the specified date and time, a second is added to the time until a unique date/time value is created.

units

Records the number of standard units of time spent upon the activity type. Where no number is specified for units, a default value of 1 unit is inserted. The total amount of time recorded against the history item is calculated by multiplying the activity's standard unit by the number of units specified in the command.

This parameter is used in conjunction with the activity parameter.

user="<user_login>"

Enables history to be saved against a user other than the current user.

user-data1 = "<NewUserDataValue>"

user-data2 = "<NewUserDataValue>"

Two additional user-defined fields can be added to the History Item dialog: User-Data1 and User-Data2. End users can enter text into these fields or the values can be specified using scripts.

,user-data parameter and S-HS-HELP scripts

When they are configured, two user-defined data fields are displayed at the bottom of the History Item dialog are with buttons to run associated scripts alongside each. For more information on configuring these fields go to User-defined History Fields.

If you have Real Time Capture or Time Sheets enabled, the User Data fields are not shown in the History Item dialog. The functionality provided by the User Data fields can only be used in documents.

Data can be written to these fields either by entering data into them directly or by using the ,user-data1 and ,user-data2 parameters in a script. The data is assigned to database fields using the HS30 and HS31 fields respectively. The User Data fields can be used to store any data values: a document reference code; guidance for a user; or, the fields can offer a user a range of valid values.

The drop-down buttons run system scripts S-HS-HELP-1 for the first field, and S-HS-HELP2 for the second. These scripts are not included in the installation of Visualfiles, developers must create them. The scripts can either overwrite the data in the field or process the data elsewhere in the system.

For example, when processing a billing cycle, the values 'approved' and 'not approved' could be offered to the user in the ,user-data1 field. Selecting 'approved' could send the record for billing and 'not approved' could forward the record to a supervisor for review. Those records with the value 'approved' could be assigned to a report of bills issued. The script to run this process would be called S-HS-HELP-1 and the values 'approved' or 'not approved' could be assigned to the field HS30.

versions

Allows multiple documents to be stored against a single history item.

This parameter is used to activate versioning on a document and is used in conjunction with the save parameter.

This functionality is used for both new and existing history items: any document stored in file history may be versioned. Versioning can be turned on or off on a single history item basis.

When Visualfiles is integrated with RM8, the creation of a new document version finalises the previous versions and prevents them from being changed.

This parameter is used in Document Bundling.

workspace=

Enables the overriding of the documents target workspace by specifying the desired workspace name. Normally documents are stored within the linked workspace to the matter for which the document is being run. Use this parameter to specify a different workspace and folder. See example 21.

Note: This parameter is only relevant when the iManage Work integration is activated.

"<MessageText>"

The specified text is displayed to the file history as the description for the history item. If MessageText is not specified, 'Unknown item' errors may be produced when the script is run.

"<AdditionalText>"

This parameter displays the specified text in the Additional Text column (the >> column in the summary view or the Notes field for an open record) in the record.

Email Parameters

attachmentcount

When using the Import Document Script to override the Save to History functionality of emails, this parameter is used to add the number of email attachments to the history item narrative.

cc

When saving emails to history, this creates a history item with a description that includes a list of people who received the email.

email-status = "<Sent|Received|Draft>"

This parameter is used to override the original status on an email being stored and gives greater flexibility when assigning an email status. So an email saved with a status of 'Received' can be set to have a status of 'Sent' or 'Draft'. This parameter is specified on a &History command within the Import Document Script.

Categorisation of emails saved to history

The default categorisation of emails is dependent upon the 'Send' event in Outlook and on the senders email address. Categorisations are independent of any mailbox or folder locations in Outlook.

The rules applied to determine which Visualfiles email category applies by default are as follows:

  • Any email saved on pressing the Send button in Outlook (i.e. the send event) is automatically saved as 'Sent'.
  • On sign in to Visualfiles the user's email address, as stored in Visualfiles User Properties is cached. Any email saved using the Save Item to History button, or via drag and drop, where the sender matches that address is saved as 'Sent'.
  • Any email saved using the Save Item to History button, or via drag and drop, where the sender is blank is saved as 'Sent'.  

Note: This condition was added specifically to work around a bug in Outlook 2003.

  • Any new email (open, prior to sending) saved using the Save Item to History button is saved as 'Draft'.
  • Any other email saved using the Save Item to History button, or via drag and drop, button is saved as 'Received'.

The Visualfiles email categorization is not intended to mirror the Outlook category structure, since it can be user -defined, and emails can be moved to, and saved from, any Outlook folder, so the source folder is irrelevant to Visualfiles.

The email-status parameter enables the default status to be overridden by logic in the Import Document script. It gives you flexibility through scripting to determine how emails are saved, overriding the default behaviour; for example if you are in an environment where a rule states that all emails from a given domain name can be considered as 'Sent' you have the opportunity to detect this.

MessageID=

Used to assign the MessageID keyword to the iManage Work email when it is stored.

Note: This parameter is only relevant when the iManage Work integration is activated.

This parameter, which is available when the iManage Work integration is used, enables the exposure of an email conversation index (unique message identifier) within the Import Document Script. See example 26.

nofileopen

Suppress the Do you want to open this file? prompt that is displayed when a user is saving emails into the file history.

recipients

When saving emails to history, this creates a history item with a description that includes the email recipients.

sender

When saving emails to history, this creates a history item with a description that includes the name of the person who sent the email.

subject

When saving emails to history, this creates a history item with a description that includes the email subject.

Example Scripts

1 Message text and additional text

In this example the (notes) modifier is used in conjunction with the input prompt to present a dialog box to the user to store Additional Text against the item.

[&History "Wrote to Other Side Solicitor";'Enter subject of letter'(notes)]

Note: Additional notes text and details can be added in the 'Enter subject of letter' section of the command.

 

Below, is a similar example that combines a field codes with text.

[&History "Wrote to " + SOL01;'Enter subject of letter'(notes)]

Note: The field SOL01 must exist in your application for this command to work.

 

This example below records two units of time for the Fee Earner carrying out the activity and records it as a Letter Out (LETO).

For this script to work the code LETO must be setup in Activity Codes, and the Fee Earner needs to have a Charge Out Rate Setup in their user record to associate a charge with the history item.

Note: Before this feature can be used you need to select the Use Time Sheet option in the File Holder screen of the User Properties dialog which adds the Time Sheet option to the navigation bar.

[&History "Wrote to Other Side Solicitor",LETO,2]

2 Noconfirm and noedit parameters

Creates a read-only item without confirmation from the user.

[&History "Wrote to Other Side Solicitor",noconfirm,noedit]

 

The noconfirm parameter provides a silent time record and does not prompt the user to confirm.

[&History "Wrote to Other Side Solicitor",LETO,2,noconfirm]

3 Save and amend parameters

Include this code in a document script or postscript of a document or a form.

This script saves the document against a History time that the user can amend using the History Item dialog, before saving.

[&History "Wrote to Other Side Solicitor",save,amend]

Note: If documents are produced in a Chain, only one save is performed. This saves the full sequence of documents as one item.

Note: If the ,save parameter is not used with a document or form then the document or form will not be stored in history, you will simple get the history text.

4 Category and save parameters

[&History "Phone Call to Other Side Solicitor",category="Solicitor"]

Include this code in a document script or postscript of a document or a form.

[&History "Test Document",save,amend]

 

We recommend that this syntax is used for the save and category parameters, where "LETOC" is a user-defined category code.

[&History "Wrote to "+SOL01;'Enter subject of letter'(notes),LETO,2,save,category="LETOC"]

For backward compatibility, the &History command still supports the previous, bracketed, syntax for these parameters, example below (LETOC is a user-defined category code):

[&History "Wrote to "+SOL01;'Enter subject of letter'(notes),LETO,2,save("LETOC")]

5 Store document in specified category

[&History "Document Test 1" ,category = "Scripts", save]

Adding a category to the command stores the generated document in the specified category and if necessary creates the category entry. Multiple categories and multi-layered categories can be created; see also &Category.

6 Add a document to multiple categories

To add a document to more than one category:

category="cat1;cat2;cat3"

This syntax adds the document to three categories.

7 important parameter

[&History "my sample item" , save , important]

8 Read parameter

The example below demonstrates reading a file whose location is stored in a database field, and turns on automatic change recording for the history item, meaning that future revisions to attached documents will automatically be saved as new versions of the document.

[&History "Database field",read=ABC01]

9 Access parameter

Asks the user how many time units should be recorded against the item, and prevents users with an access level lower than 10 accessing the history item.

[&History "File Opened",LETI'Enter number of standard units to charge'(number),access=10,]

10 Level parameter

Allows only Supervisors and above in the User Hierarchy to access a history item.

[&History "File Opened",level="supervisors"]

11 Subject, sender, recipients and cc fields

This example creates a history item with a description that includes the email subject, who sent it and a list of people who received it; used in the Import document Script on an application.

[&History "Email: " + subject + ", from " + sender + ", to " + recipients + ", ccd to " + cc]

Note: An email can be identified in the Import document script by using a condition for example, &If DOCTYPE = "E".

12 Copy timer in RTC for a file to history

This command will copy any timer currently running in Real Time Capture for a case to the History screen, rounding up to the value to the next standard unit.

Note: The amend parameter must be added whenever the rtc parameter is used in the post script of a document.

[&History "Letter to client", save, rtc, amend]

Note: The activity code is not copied over. The timer is then deleted.

13 Preview parameter with iManage Work

[&History, "iManage Work Test",preview]

14 Time recording

Time recording onto a file from a script to record time for the last working day.

[&Assign LF01='Please enter number of time units']

[&Assign LF02='Please enter description']

[&Assign LF03='Please enter notes']

[&Assign LF04=date-1(working day)]

[&History LF02;LF03,p,LF01,amend,date=LF04]

15 Create a new history record

[&History subject ,read=file,save]

16 Create a version of an imported document silently

[&History,read=file,save,versions,docid=docid,auto-save]

Or for an email:

[&History subject,read=file,save,docid=docid,versions,auto-save]

17 Overwrite a version of an imported document silently

[&History ,read=file,save,docid=docid,auto-save]

Or for an email:

[&History subject,read=file,save,docid=docid,auto-save]

18 Create a version of an imported document

[&History,read=file,save,versions]

Or for an email:

[&History subject,read=file,save,docid=docid,versions]

Note: When this script is run the 'Version' prompt appears.

19 Overwrite a version of an imported document

[&History,read=file,save,auto-save]

Or for an email:

[&History subject,read=file,save,docid=docid,auto-save]

Note: When this script is run the 'Save' prompt appears.

20 Key Date

Used in Document Bundling.

[&History "CoverDocument" , save , category="Bundle",keydate="01/01/2005"]

21 workspace parameter

Use the workspace parameter to specify a different workspace and folder; for example, save the document in the folder folder1 under the workspace of Workspace2:

[&History "doc1", category="folder1", workspace="Workspace2",save]

Note: If more than one workspace exists with the specified name then the document is rejected.

If the workspace parameter is omitted then the document is saved to the default target workspace as before.

Note: This parameter is only relevant when the iManage Work integration is activated.

22 metadata parameter

[&History "Create metadata document", metadata, save]

23 call parameter

It is currently possible to add prescripts and postscripts to a precedent document in Visualfiles. However, when a document is produced by the end user, first the prescript is processed, then the code in the document itself, then the postscript and finally the document is presented to the user to edit. The only command Visualfiles runs after the document has been closed is the &History command. Therefore it was not easy to capture information to be stored after the document was edited by the end user. So in order to profile a document after the document has been stored Call=scriptname can be added to the &History command which allows you to action a script once the history item has been created and to capture metadata that is stored against the document after the user has completed drafting their document.

[&History "Doc Save Test", call=test-script,save]

If this &History command is assigned to a document that is then actioned, when all processing is complete the script test-script is run.

test-script syntax:

[&Select HS:last]

[&MetaData history, create]

[&Assign HS99(1)="Metadata 1"]

[&Assign HS99(2)="Metadata 2"]

This script is run after the document is complete and attaches a metadata screen to the history item with the first two metadata fields populated.

The advantage of this is that when all processing is complete the history is available for scripting which then enables metadata to be attached to the history item. See Document Metadata.

24 Import documents to a file, read parameter

[&History "document_name",read=file]

25 File name supplied to ,read parameter

[&Assign LF02='Select file...'(filename)]

[&Import filename=LF02,delimiter=","]

 

[&EndImport]

[&History "Imported File",save,noconfirm,read=LF02]

26 Import Document Script with the MessageID assigned

[&Assign LF01=Subject]

[&Assign LF02=MessageID] // MessageID Keyword

[&Assign LF04="E-Mail"]

 

[&Assign LF07="substring('"+ReceivedTime+"',1,19)":Proval ]

[&Assign LF08="substring('"+SentTime+"',1,19)":Proval ]

 

[&History LF01, Save, NoConfirm, Category=LF04, Class="E-MAIL", Profile="no", Custom13=Sender,Custom14=Recipient,Custom21=LF07,Custom22=LF08,Custom23=LF07,Custom24=LF07,MessageID=LF02]

When the &History command is encountered the iManage Work integration first checks if an email with the specified email has not already been assigned under the target workspace. If it has already been stored then the import is rejected and messaged to the user. If no email exists with the MessageID then it is stored as normal with its MessageID.

This mechanism allows the rejection of potential duplicate storage of emails under a workspace.

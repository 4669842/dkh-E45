&SelectRelation

Syntax    

[&SelectRelation type = <RelationCode>]

Example Scripts

Description

Selects a relation from the database.

Note: If you want to select a relation using a relation- field where the relation-field UD code is held in an entity, this is possible using the &SelectEntity command. For more information go to Selecting an Entity from a Relation Field.

Parameters

always-show

Ensure that the Finder is opened, even if there is only one matching relation.

Usually, if the relation described by the command matches only one relation, then this relation is selected, the target of that relation becomes the default owner and the script continues without opening the Finder. However, if the always-show parameter is included in the command, then the Finder is always opened, regardless of the number of matching relations.

button<X> = "<Button<X>Text>"

Assign text to buttons displayed in the Finder. The &SelectRelation command can display a dialog with up to five buttons: the buttons are named button1, button2, button3, button4 and button5. To have one of these buttons show on a screen, say button3, add button3 = "button 3 text", to your command.

contains =

Returns results where the description includes the value specified in this parameter.

count = <DBCode>

Return a count of the number of related items.

default = "<ButtonNumber>"

Assigns a button as the default. The default button has the focus applied and its value will be assigned if a user simply presses Return on a screen. If the default parameter is not used, focus will apply to button1. Valid values are 1, 2, 3, 4, and 5, representing each of the five buttons.

fetch = "<All|Numeral>"

Enables a developer to control how many records will be returned to the Finder.

first

Selects the first relation

grid-relation-field

The process that fills a grid, notes which fields are relation fields and returns data into the grid in a format that the &SelectRelation command can resolve easily. The grid-relation-field parameter is used to pass in the coded data from the grid.

noadd

Remove the Create button from the Finder, preventing users from creating new items when selecting a relation.

ref = "<TextString>"

The ref parameter is used to give the located target a reference.

When specified, the ref parameter provides a quick reference to the selected relation/target that can be used throughout the remainder of the script.

relation-field = <DBCode>

Where a screen has a field of type Relation, the relation displayed in the field may be selected via the relation-field parameter. See example 9.

response = <Field>

Specify the location to which the user response should be stored.  

If the user double clicks on their selection or highlights a finder entry and presses return, response will equal "GO".  If the user clicks on the close window button in the top right hand corner, or presses escape, response will equal "CANCEL".  If any of the user-defined buttons are clicked, response will equal "BUTTONX".

title = "<Title>"

Assign a title to the dialog box.

type = <RelationCode>

The type of relation to be selected.

This command only works if the default owner is an entity that is able to hold  that relation and that the relation is pre-existing. The target of the relation then becomes the default owner. This is not allowed in batch scripts, that is, from the To-Do List or the Robot.

The concept of 'default owner' is fundamental to scripting in Visualfiles. The default owner is the file, entity or role in which a script is located at a given point in time during the processing of the script. The default owner changes as the script moves through the elements in the system.

System scripts, by nature, have no default owner and  will inherit the last accessed component as the owner as the script progresses.

For example, in very simple terms, say you have a  system script that starts at an entity then references a role and, finally, selects a file.

As the script is processed and the role is referenced, ownership is transferred to that role and it becomes the default owner. Similarly, as the script is processed and the file is selected, the file will be the default owner.  

The default owner for all other scripts will be the file, entity or role  where the processing starts, but as with system scripts this can be changed using the &With command.

Status (SelectRelation)

Whenever an &SelectRelation command is executed an internal status is created.  If the command has been successful then the status is blank. If the command has been unsuccessful, a text message will be assigned to the status. Therefore, testing the status of an &SelectRelation command can be used to check whether a relation exists.

Syntax    

[&SelectRelation type = "<relation code>"]

[&If Status(SelectRelation)<>""]

   [&CreateRelation type = "<relation code>"]

[&EndIf]

Example Scripts

1 Count parameter

This code will return the number of related items and place this number into TF01.  To ensure that the correct value is determined, set TF01 to 0 before the count.

[&Assign TF01 = 0]

[&SelectRelation type = "RCLI_RREP",count = "TF01"]

2 No-add parameter

This code will display the Finder without the Create Entity button.

[&SelectRelation type = "RCLI_DEBT,noadd]

3 Fetch parameter

This code will display the first 3 matching records in the Finder. If the fetch parameter was assigned the value all, it would display all the matching records in the Finder.

[&SelectRelation type = "INTRO_CLIENT",button1 = "Add",button2 = "Select", title = "Select Or Add Introducer",fetch = 3]

4 Default parameter

This code assigns the focus to Button2 when the screen is initially displayed. If the default parameter was not used, the focus would apply to Button1.

[&SelectRelation type = "INTRO_CLIENT"",button1 = "Cancel",button2 = "Add",button3 = "Select",title = "Select Or Add Introducer",default = "2"]

5 Performing actions on the basis of user response

This example uses the &SelectRelation command to present the user with a Finder for the current individual displaying their employments.  The four buttons allow the user to cancel the action, set the selected employment as the individual’s principal employment, add a new employment, or simply open the one selected.

[&SelectRelation type = "EMPL_EMPL",button1 = "Cancel",button2 = "Set as principal",button3 = "New",button4 = "Open",title = "Employments for "+GivenIndi.Nam11,default = "4"response = TF01,ref = "givenEMPL"]

[&If TF01 = "button2"]

   ** set as principal employment by creating relevant relationship

   [&CreateRelation type = "PRIEMPL_EMPL",to-ref = "givenEMPL"]

[&Else]

   [&If TF01 = "button3"]

      ** create new employment

      [&CreateRelation type = "EMPL_EMPL"]

   [&Else]

      [&If TF01 = "button4" &Or TF01 = "GO"]

         ** open selected employment

         [&Show givenEMPL.EMP]

      [&EndIf]

   [&EndIf]

[&EndIf]

6 Status

This code checks to see if the default owner has been related to a customer, that is, is the employer of this person known? If the status of the &SelectRelation command is not blank, that is, the relation does not exist, then the user will be prompted to create it.

The status of an &SelectRelation command is also useful as a troubleshooting tool. If an action containing an &SelectRelation command is not providing the required output, then the developer may use the following structure to try and find out the cause of the problem.

[&SelectRelation type = "CONTACT_CUSTOMER"]

[&If Status(Select)<>""]

   [&CreateRelation type = "CONTACT_CUSTOMER"]

[&EndIf]

7 Status

[&CreateRelation type = "CONTACT_CUSTOMER"]

[&Message Status(&CreateRelation)]

If this script was within the agenda of entity type, Individual, the message may show a status of Relation not valid for selected item. In this instance, the status indicates that the individual in question does not have a role of Contact.

8 Populate a grid with grid-relation-field

The relation fields can then be tested against.

[&Foreach GR, GR01<>"",noincludeall]

[&Selectrelation grid-relation-field=GR01,ref="myCORRES"]

   [&If myCORRES.EN06="COMM"]

      [&Assign LF01=LF01+1]

   [&Endif]

[&Endfor]

9 relation-field parameter

[&SelectRelation relation-field=FIL01,ref="myfield"]

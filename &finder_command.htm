&Finder

Syntax    

[&Finder]

Example Scripts

Description

Opens the Finder (Quick Steps) and enables search definitions to be added to it, such as predefined queries built through the Query Wizard.

&Finder enables you to return a simple list of files or entities for a defined list of types and return that data to the screen (default) or to an xml field, a grid or a pipe-delimited text file.

&Finder can be used in a similar way as the &Grid command; buttons can be added, user responses stored and internal codes selected, using the response and result parameters.

Note: If you are assigning a Local Field (LF) to pass through to a query, ensure that the LF you add in the Query Criteria is surrounded by square brackets i.e. [LF01].

Parameters

autoexport

When this command is used to action a query and you use the existing autorun option, Visualfiles executes the query but does not perform the export associated with the query.

When the autoexport option is added along with autorun, the query executes and the data is exported without prompt. See example 12.

autorun

Runs the query specified by the ,runquery parameter so that the Finder displays the results automatically. If this parameter is not included, the user will be asked to confirm all details of the query.

begins=

Sets the search criteria to be “return records where the description begins" the value you enter.

button<x> = "<LabelForNoSelectButton>"

Sets the name for button1 to button5 displayed in the Finder.

contains=

Sets the search criteria to be “return records where the description contains" the value you enter.

dbfields

Can be used with the output="grid" or output="file" options to return additional database fields to your result set.

doc-only

Enables the Finder to be called with all search criteria removed except document searching ability.

entity-only, type="<Entity Type>"

Enables you to search only on entities.

You can use either "entity-only" or "file-only" in any one &Finder call.

Note: Use the syntax No-Entity in combination with the entity-only parameter to specify the role code and entity type. See example 11.

Note: If you encounter performance problems when using the entity-only parameter try using a query instead. Create a query in the Finder then reference it in your script, for example:
[&Finder runquery="MatterGroup",autorun,return-all,output="grid"]

fetch = "<{Numeral|All}>"

Enter a numeric value for that number of records to be returned or use the word “all" to return all records to the Finder.  

Leaving this off your command returns a maximum of 50 records.

filename = "<filename>"

Used with the output="file" option to specify the directory and name of the text file to create.

file-only, type="<File Type>"

Enables you to search only on files.

You can use either "entity-only" or "file-only" in any one &Finder call.

levelcode = "level"

Set the fee earner level for this query. See example 13.

Tip: The levelcode for the 'Entire Organisation' is "SYS".

nodisplay

This parameter is used to prevent the Finder from appearing if 1 or 0 results are found.

no-save

Removes the Save Current Query, Open a Saved Query and Delete a Saved Query icons from the Query Builder, preventing the user from modifying or loading the queries.

If the query specified in the runquery parameter is a private query to which the current user does not have access, the no-save parameter will be added automatically by the system.

Note: This parameter only applies to the Quick Steps Finder also known as the floating or popup Finder. Only this version of the Finder can be called using &Finder.

output

Specifies the output for the results of the search.

If this option is not included in a command, the results are displayed in the normal Finder.

output="grid"

Transfers the output to a grid which you can then refer to in your script using a “foreach GR" – the Finder is not displayed.

The grid will contain equivalent columns to those that would have been displayed in the Finder itself with the addition of the int-code for each result.

output="file"

Redirects the results from the Finder to a pipe delimited text file. The Finder is not displayed using this option.

output="file",filename="<filename>"

Must be used with the output="file" option to specify the directory and name of the text file to create.

output="grid",dbfields="<database fields>"

or

output="file",dbfields="<database fields>"

Can be used with the output="grid" or output="file" options to return additional database fields to your result set. This is a comma delimited list of database field codes.

output="<xml field>"

If you use "xl??" then the results of the Finder are transferred to that xml field which you can then refer to in your script; the Finder is not displayed.

queryonly

Prevents access to anything but the query specified by the ,runquery parameter.

As with the no-save parameter, the Save Current Query, Open a Saved Query and Delete a Saved Query icons are removed from the Query Builder. The Query Builder lookup button within the Finder is also disabled.

Note: This parameter only applies to the Quick Steps Finder also known as the floating or popup Finder. Only this version of the Finder can be called using &Finder.

response = <FieldName>

Specifies the location to which the user response should be stored. See example 9 & 10.

result = <FieldName>

The internal code of the selected item. See example 9 & 10.

return-all

Executes the query and returns all the results to the script without displaying the Finder.

runquery = "<QueryName>"

Loads a query with the name specified: a literal query name or a database code. If no query is specified the Finder will simply be presented to the user.

saved-index

Used in conjunction with the file-only or entity-only parameters to specify that the saved index field criteria should be used.

show-file-owner

Used in conjunction with the file-only option to display a File Owner column in the Finder results.

status = "<Open|Complete|All>"

Used to set the status of files to search and return either Open, Complete or All (Open and Complete).

title"<TitleFinder>"

Sets the title of the Finder.

type=

Restricts the Finder to a certain set of file or entity types, depending on the use of entity-only or file-only.

If you want to restrict the types used in the Finder, then provide a comma delimited list of file or entity-type codes. To use this option you must use either "entity-only" or "file-only" in your [&Finder] call.

Note: When the type= parameter is specified it overrules the default values for the Finder.

word=

Sets the search criteria to be “return records where the description has the word" that you enter.

xmlresult = <xlnn>

Specifies the XML document where selections from the Finder will be recorded.

Example Scripts

1 Prepare the Finder to run a specified query

[&Finder runquery="Potential Clients"]

2 Specify a query and run the Finder, returning the results to an XML file

[&Finder runquery="Potential Clients",autorun,title="Potential Clients",xmlresult=xl01]

3 Record the results from the query without showing the Finder

This example will return all the results from the Potential Clients query into xl01 without displaying the finder

[&Finder runquery="Potential Clients",autorun,return-all,title="Potential Clients found",xmlresult=xl01]

4 Default buttons

In this example a query is automatically run with user-defined button titles and a default specified in the expectation that no result will be found.

[&Finder xmlresult = xl01, runquery = "Find Complaint" ,autorun ,button1 = "Select Highlighted Issue" ,button2 = "Not Found" ,title = "Matching Complaints" ,default="2"]

5 Return all sale files into a pipe delimited text file, returning the addition fields ABC05 and XYZ10 from each of the resulting files

[&Finder file-only, type="sale", output="file", filename="c:\salefiles.spr", fetch="all", dbfields="abc05,xyz10"]

The above example has created a text file with an extension .spr. This can be used in conjunction with the system option “202 – Microsoft Excel" and some options on the [&open] command to allow the resulting data to be opened and formatted in excel:

[&open excel="c:\salesfiles.spr"]

will open the resulting data in an excel spreadsheet.

You can use the syntax:

[&open excel="c:\salesfiles.spr", options="autofilter"] to enable autofilters in the resulting spreadsheet for sorting of the excel data.

6 Create a .spr file opened in Excel, auto-formatted with filter options

[&Finder file-only,type="AGEN",WORD="banking",output="file",filename="C:\dbtest.spr",dbfields="MAT11"]

[&Open excel="c:\dbtest.spr", options="autofilter"]

7 Find files of type sale or purchase that have the word Johnson in the description

[&Finder file-only, type="sale,purc", fetch="all", word="Johnson", output="grid"]

[&ForEach GR, noincludeall]

  [&SelectFile int-code=GR09, ref="MyFile"]

  /*do some work */

[&EndFor]

8 create a .spr file using &Finder and open in Excel, then auto format and add filter options

[&Finder file-only, type="AGEN",WORD="banking",output="file",filename="C:\dbtest.spr",dbfields="MAT11"]

[&Open excel="c:\dbtest.spr", options="autofilter"]

9 Open the Finder, search matters and display screen

The following script displays the Finder, searching for all matters of type MATA. The internal code of the selected file goes into LF02 and the name of the button/response goes into LF01. Then you can select the file and perform any work required. Pressing Select here opens a user-defined screen, PYR, which resides on the file. Otherwise selecting New, creates a new file:

[&Finder file-only,type="MATA",button1="Quit",button2="Select",button3="New",response=LF01,result=LF02]

[&If LF01="BUTTON1"]

[&Quit]

[&Else]

[&If LF01="BUTTON2"]

[&Selectfile int-code=LF02,ref="myFILE"]

[&With myFILE]

[&Show PYR]

[&EndWith]

[&Else]

[&If LF01="BUTTON3"]

[&CreateFile type="MATA"]

[&Endif]

[&Endif]

[&Endif]

10 Searching for a file/matter using MT05

The Query “QueryTest” has search criteria set as all applications, field MT05 (file code).  The operator is contains/begins/equals [TF01]

Note: You must use square brackets when defining query temporary field otherwise system reads as a text input.

TF01 is then set is the system script below prior to running the query. This example allows the file to be selected on double click or using a select button.

[&Assign TF01 = 'Please enter the file code you wish to search for']

[&Finder runquery = "QueryTest",autorun,button1="Cancel",button2="Open",response=TF02,result=TF03]

[&If TF02="button2" &Or LF03="GO"]

   [&SelectFile int-code=TF03, ref="myFile"]

   [&With myFile]

      [&Open]

   [&EndWith]

[&EndIf]

11 Role selection only

Use the following code, which includes the No-Entity switch, to specify the role code, mailshot.

[&Finder entity-only, No-Entity, options="role-codes,mailshot"]

Use the following code to specify the role code mailshot, and entity type.

[&Finder entity-only, No-Entity, options="role-codes,mailshot,entity-type-codes,mail"]

12 autorun parameter

[&Finder runquery="MyExcel", autorun, autoexport]

13 Locate all individual entities for the current user - levelcode parameter

[&Assign LF01 = US10 + "/" + US01]

[&Finder entity-only,output="xl03",word="jones",type="indi",levelcode=LF01]

Setting the level in the script overrides any setting currently saved against the user's settings  used as a standard Finder default.

Developers' Tips

User Access

In addition to the queryonly and no-save parameters, there are other issues regarding security and a user’s access to queries.

Users without access to the Query Wizard in the Finder or specific queries can still use scripts which access the Finder. In addition, if the user does not have access to the entities/applications/roles being searched, they can still run the query, even though they would not normally be able to do so from the Finder. This is allowed because the entity/file cannot be opened from the Finder when called from the script.

If the user does not have Query Wizard access rights assigned they cannot access it from the Finder when it is called from a script. However, they can still run the query and return the results to the script.

To prevent access to other queries, use the queryonly parameter which suppresses all other options in the Finder.

XML Result

The format of the XML document is:

<MAIN>

<ITEM type="ENTITY" or "FILE">int-code</ITEM>

<ITEM type="ENTITY" or "FILE">int-code</ITEM>

</MAIN>

There is an ITEM element for every returned file or entity. Click here to view a sample script to decode this XML document, and how to use the int-code which is returned.

The following script displays the Finder, searching for all matters of type MATA. The internal code of the selected file goes into LF02 and the name of the button/response goes into LF01. Then you can select the file and perform any work required. Pressing Select here opens a user-defined screen, PYR, which resides on the file. Otherwise selecting New, creates a new file:

[&Finder file-only,type="MATA",button1="Quit",button2="Select",button3="New",response=LF01,result=LF02]

[&If LF01="BUTTON1"]

    [&Quit]

[&Else]

    [&If LF01="BUTTON2"]

        [&Selectfile int-code=LF02,ref="myFILE"]

        [&With myFILE]

            [&Show PYR]

        [&EndWith]

    [&Else]

        [&If LF01="BUTTON3"]

            [&CreateFile type="MATA"]

        [&Endif]

    [&Endif]

[&Endif]

 

This script prompts for search criteria, stores the criteria in the temporary field, TF01, then runs a query which refers to TF01. It then creates a case for every entity returned (unless one already exists), and updates some data on the case.

 

It makes use of &Finder, &Assign, &Select XL, &DoWhile, &If and &Message commands.

[&Assign TF01='Enter search criteria for clients']

[&Finder xmlresult=XL01,runquery=clients,autorun]

 

/* Select the “main" element – for example, the document */

[&Select XL,Type=XL01,Element=Main,parent]

 

/* Select the first “item" element, and set it as the parent */

[&Select XL,Type=XL01,Element=item,first,parent]

[&Assign TF01 = 0]

 

/* Loop until the select command has generated an error */

[&DoWhile status(select) = "",includeall]

 

/* Set TF01 to the value of the current element, for example, the int-code */

   [&Assign TF01=XL01(element=item,current)]

 

/* Get the entity using the new int-code option, and create a reference */

   [&SelectENTITY,int-code=TF01,ref="test"]

 

/* Select the related usergroup case for this referenced entity*/

/* If it fails, create the relation, and therefore the case */

   [&SelectRelation usergroup_client,from-ref="test"]

   [&If status(SelectRelation) <> ""]

      [&SelectRelation usergroup_client,from-ref="test",create]

   [&EndIf]

 

/* Set the description, and a UD Field MEE01 */

   [&Assign test.usergroup_client.mx01= "AGM for " + test.en02]

   [&Assign test.usergroup_client.mee01=test.en02]

 

/* Select the next Item element, and set this as parent */

   [&Select XL,Type=XL01,Element=item,next,parent]

[&EndDoWhile]

[&Message "Created meetings"]

 

/* Run another query which shows all the usergroup cases currently on file */

[&Finder runquery=usergroups,autorun]

Begins/Word/Contains

The contains= option should only be used as a last resort; it will be slow compared to “word=" and “begins="

Also note, you can only use one of begins/word/contains in any one &Finder command.

Query Builder Access Right

Provided the &Finder command contains the autorun and queryonly parameters, a query can be run from the &Finder command when users do not have the Query Builder access right selected. The query is displayed in the Finder but the user is not be able to access any other queries.

The following syntax would result in the query Matterquery being run:

[&Finder runquery="Matterquery", queryonly,autorun,title="Matter Query"]

If either the queryonly or autorun parameters were missing the normal Finder is displayed with no link to the query.
